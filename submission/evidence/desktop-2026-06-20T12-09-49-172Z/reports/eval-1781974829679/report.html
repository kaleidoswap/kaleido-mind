How to read this

The question: the same wallet capabilities are offered to each model three ways. Higher = the model picked the right tool more often. We grade the model's first decision (one inference, no execution) and run each case 3× for confidence.

Mechanisms

  • fc — function calling: a few curated tool schemas. The clean baseline.
  • mcp — the same tools plus ~46 decoys (≈60 total), like a real MCP server. Tests selection under a large surface.
  • skill — a skill narrows the tools to ~3–9 and injects a playbook (our portable default — works on mobile, where CLI doesn't exist; on desktop a skill can also drive a CLI).

Metrics (per cell)

  • % — task success across all repeats: right tool ✓ and right args ✓. The headline.
  • % rel — reliability: share of cases that passed on every repeat (consistency, not luck).
  • sel — picked the right tool (ignoring arguments). args — got the arguments right when it picked right.
  • ms — average decision latency (model resident in RAM). trials = cases × repeats.
  • ⚠ — over-trigger: called a tool on a greeting (should have just replied). Lower is better.

Categories: wallet, trading, commerce (Bitrefill), knowledge (explain → search), memory (remember/recall), negative (greetings → must NOT call a tool).

No execution — we grade the model's chosen tool call directly, against a seeded dataset, with the model loaded once and reused. Reproducible; measures decision quality.
